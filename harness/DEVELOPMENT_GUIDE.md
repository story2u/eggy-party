# Development Guide

## 代码风格

- 保持简单 Python 代码，避免为 demo 引入复杂框架。
- 优先沿用现有类和函数结构。
- 变量名保持直观：`x`, `y`, `vx`, `vy`, `platforms`, `obstacles` 这类游戏语义可保留。
- 注释用于解释游戏规则、坐标约定或非显然逻辑，不要逐行解释简单代码。
- 新增中文 UI 文案时确认字体可渲染，使用 `get_font(size)`。

## TDD 规范

- 新功能先写测试，测试名描述行为，docstring 描述设计意图。
- 修 bug 先写能复现 bug 的回归测试。
- 测试按模块放在 `tests/test_*.py`，优先使用 Python 标准库 `unittest`。
- 不要只测试实现细节；优先测试玩家能感知的行为和模块公开约定。
- 改变需求时可以改测试，但必须保留新的意图说明。

## 坐标和物理约定

- Pygame 坐标系：左上角为 `(0, 0)`，向右为 x 正方向，向下为 y 正方向。
- 玩家位置 `x, y` 表示角色矩形左上角。
- 平台数据格式固定为 `(x, y, w, h)`。
- `dt` 使用秒，不要混用毫秒。
- 向上跳跃速度为负值。

## 模块边界

- `game.py` 可以编排对象，但不要把所有对象内部逻辑都塞进去。
- `player.py` 负责玩家和 AI 自身行为。
- `obstacles.py` 负责障碍自身状态和绘制。
- `level.py` 只放关卡数据，避免放运行时逻辑。
- `font_utils.py` 只负责字体发现和创建。

## 新增功能建议

新增障碍时：

1. 在 `obstacles.py` 新增类。
2. 保持 `update(dt)`、`draw(screen, camera_x)`、`get_hitbox()` 接口。
3. 在 `level.py` 的目标关卡中实例化。
4. 如果涉及碰撞效果，在 `Game._update_playing()` 中统一接入。

新增关卡时：

1. 在 `level.py` 创建 `LEVEL_N`。
2. 设置 `spawn_x/spawn_y`、`finish_x/finish_y`、`ai_spawns`。
3. 加入 `ALL_LEVELS`。
4. 更新 `harness/FEATURE_MAP.md`。

新增 UI 文案时：

1. 尽量使用普通中文和 ASCII，谨慎使用 emoji。
2. 使用 `font_utils.get_font` 创建字体。
3. 确认文字不会超出 HUD 或画面边界。

## 不建议的改动

- 不要为了一个小 bug 全面重写主循环。
- 不要把关卡数据改成复杂 DSL，除非后续确实要做地图编辑器。
- 不要直接依赖某个机器专有字体名。
- 不要新增必须联网才能运行的逻辑。
- 不要把临时调试输出长期留在主循环里。

## 文档同步规则

代码变化触及以下内容时，必须同步 harness：

- 文件职责变化。
- 新增/删除关卡、障碍、玩家能力。
- 新增依赖或运行命令。
- 新增测试方式。
- 改变长期架构方向。
